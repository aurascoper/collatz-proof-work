# Cycle Exclusion Engine — initial benchmark report

**Date**: 2026-04-28
**Hardware**: Apple M-series, single-thread (`cargo build --release`).

## Phase 1 (Rust) — generation + composition

| K  | records F_{K+2} | total | rec/s gen+compose | rec/s with NDJSON write |
|----|----------------:|-------|-------------------|--------------------------|
| 12 | 377             | 1 ms  | 2.8 M/s           | 2.9 M/s                 |
| 24 | 121,393         | 0.07 s| 1.7 M/s           | 2.1 M/s                 |
| 36 | 39,088,169 (F_38) | 41 s | ~1.0 M/s         | n/a (bench-only)        |

**Observation**: BigInt allocation per block is the bottleneck above
K~24; small-int (i128 or u128) fast path could 5x throughput at K ≤
~70.

## Phase 2 (Julia) — pipeline classification + verification

End-to-end at K=24:

```
  NON_INTEGER_FIXED_POINT     118,009  (97.2 %)
  NEGATIVE_OR_ZERO_DENOM        3,380  ( 2.8 %)
  REALIZABLE_TRIVIAL                1            (n = 1)
  REALIZABLE_CANDIDATE              2            (n ∈ {2, 4})
  NON_POSITIVE_FIXED_POINT          1

                              121,393 total

Non-trivial Collatz cycles    0            ← confirmed
Trivial-cycle rotations       3            ← exactly the 1-2-4 cycle's 3 elements
```

Julia runtime: 2.5 s including JSON parsing, BigInt division, and full
direct-simulation verification of the 3 candidates.

## Comparison to Iteration 069a

| Metric                  | Iter 069a (K=8)             | CEE K=24                  |
|-------------------------|-----------------------------|---------------------------|
| Constraints / blocks    | 12,711,807 LP edges         | 121,393 parity blocks      |
| Tools used              | scipy HiGHS + Fraction      | Rust BigInt + Julia BigInt |
| Result                  | 0/12.7M LP failures         | 0 non-trivial Collatz cycles |
| Runtime                 | 80s enum + 219s exact verify | 0.2s gen + 2.5s verify    |
| Rigorous?               | Yes (closed graph)          | Yes (single-window cycles only) |

The CEE pipeline is **complementary** to 069a: the LP gave a
cycle-negativity certificate at the *abstraction* level (closed K=8
graph); the CEE gives an *integer-realisability* statement at the
parity-block level. Both are honest, both are weaker than Hercher 2022
in their current form.

## Next steps (Phase 1.5 / 2.5)

- Phase 1.5: parallelise Rust generator with `rayon`; expected 8-10x
  speedup on M4 → K=40 in seconds, K=44 in minutes.
- Phase 1.6: add binary `bincode` writer for K > 36 batches.
- Phase 2.5: replace `baker_lower_bound` with explicit constants from
  Laurent / Mignotte; this is the gating item for any rigorous claim
  beyond direct enumeration.
- Phase 3: multi-window cycles (length L ≥ 2) via
  `ParityBlock::compose`; here the artefact-cycle family explodes
  combinatorially but should be tractable up to L*K ≤ 100.

## What is proven by this prototype

The pipeline reproduces, at K=24 windows, the well-known fact that
the only realisable Collatz cycle of period ≤ 24 is the trivial
1-2-4 cycle. Per `program.md`, this is **not** a claim about Collatz
in general; it is a re-verification at the level of single-window
parity composition.
